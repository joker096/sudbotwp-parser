# AGENT RULES

## Режим работы
- Выполняй задачу до конца без пауз и остановок.
- Не спрашивай подтверждение перед обычными действиями.
- При неясности делай разумное предположение и продолжай.
- Показывай только короткий прогресс и финальный результат.

## Автономность
- Всегда сначала анализируй локальный контекст (файлы, код, конфиги).
- Если есть несколько путей, выбирай самый безопасный и быстрый.
- Не останавливайся на плане: сразу переходи к выполнению.

## Ограничения безопасности
- Не выполняй разрушительные действия (удаление данных, reset/hard, массовые чистки) без явной команды пользователя.
- Не раскрывай секреты из env/ключей в ответах.

## Качество результата
- После изменений делай быструю самопроверку (сборка/тесты при наличии).
- В финале давай: что сделал, какие файлы изменил, что проверить.

## Концепция универсального механизма для OpenCode: Интегрированный подход

На основе всестороннего анализа проблем, стратегий и архитектурных решений, представленных в источниках, можно сформулировать концепцию универсального механизма для OpenCode. Эта концепция объединяет лучшие практики в единый, последовательный и гибкий процесс, предназначенный для непрерывного исправления кода вне зависимости от размера кодовой базы и сложности задачи. Механизм состоит из трех ключевых этапов: инициализация и планирование, итеративная обработка сегментов и агрегация с финальной реконструкцией.

**Этап 0: Инициализация и планирование.** На этом начальном этапе OpenCode принимает на вход задачу (или несколько задач) и исходный код, который необходимо обработать . Задача может быть сформулирована как текстовая инструкция (например, "исправить уязвимость XX в файле Y.py") или как более сложный запрос, требующий анализа нескольких файлов. Система начинает глубокий статический анализ всей подлежащей исправлению области кода. Этот анализ включает построение дерева синтаксического анализа (AST) для изоляции логических блоков (функций, классов), а также проведение анализа потоков данных (DFA) и управления (CFA) для построения полной карты взаимосвязей между этими блоками [[2](https://arxiv.org/html/2502.07049v2)]. На основе этого анализа и описания задачи система формирует детальный план действий. Этот план представляет собой последовательность шагов, которые система будет выполнять. Он включает в себя список всех сегментов кода (например, функций), которые необходимо проанализировать, а также их правильный порядок обработки, определенный с помощью топологической сортировки по графу зависимостей [[2](https://arxiv.org/html/2502.07049v2)]. Это гарантирует, что система будет обрабатывать сначала независимые компоненты, а затем те, от которых они зависят, предотвращая проблемы с неверными ссылками на еще не исправленный код.

**Этап 1: Итеративная обработка сегментов.** Этот этап является ядром механизма и реализует гибридную архитектурную стратегию, сочетающую агентное планирование с детерминированным рабочим процессом [[6](https://www.linkedin.com/posts/niels-rogge-a3b7a3127_when-people-ask-me-which-framework-should-activity-7333773142873591809-vqUc)]. Для каждого сегмента кода в заранее составленном плане система выполняет серию повторяющихся действий:
a. **Формирование контекста:** Это самый критический шаг. Для текущего сегмента система использует программное разрезание, чтобы извлечь минимально необходимый фрагмент кода, который напрямую связан с местом, где должна произойти корректировка [[2](https://arxiv.org/html/2502.07049v2)]. В этот контекст добавляются несколько ключевых элементов: само разрезанное тело кода, подробное описание задачи (например, из CWE/CVE), примеры успешных исправлений для похожих проблем (если доступны), и, что очень важно, информация о уже выполненных изменениях, сохраненных в реестре [[3](https://www.mdpi.com/2504-4990/7/4/149), [4](https://dl.acm.org/doi/10.1145/3708522)]. Это обогащенное знаниями и контекстом сообщение отправляется в LLM.
b. **Вызов LLM:** Сформированный контекст отправляется в LLM через прямой API-вызов. Полученный ответ (исправленный код, комментарий, найденная новая уязвимость и т.д.) считывается и анализируется.
c. **Обновление состояния:** Полученный результат немедленно сохраняется в внешнем хранилище состояний (например, базе данных или файле). Этот шаг является фундаментальным для обеспечения непрерывности. Он позволяет системе "помнить" свои действия и продолжать работу после возможных сбоев. Также обновляется внутренняя карта зависимостей, если обработанная функция теперь имеет новые свойства или уязвимости, которые могут повлиять на другие части системы.

**Этап 2: Агрегация и финальная реконструкция.** После завершения обработки всех сегментов согласно плану, система переходит к финальному этапу. Здесь происходит сбор и синтез всех промежуточных результатов. Система собирает все исправленные фрагменты кода, агрегирует комментарии и отчеты об уязвимостях. Затем выполняется финальная проверка и слияние изменений, чтобы убедиться в их совместимости и отсутствии конфликтов. Например, если две независимые функции меняли одну и ту же глобальную переменную, система должна убедиться, что их изменения не противоречат друг другу. По итогам этого этапа OpenCode генерирует итоговый исправленный код, который представляет собой исходный код с внесенными изменениями, а также подробный отчет о всей проделанной работе: какие задачи были выполнены, какие уязвимости были найдены и исправлены, и какие изменения были внесены в код.

Этот интегрированный подход обеспечивает универсальность, необходимую для решения широкого круга задач по исправлению кода . Он не привязан к конкретному типу операции (рефакторинг, исправление ошибок, поиск уязвимостей), так как его ядро — это гибкий механизм обработки сегментов, который может быть адаптирован под любую задачу путем изменения контекста, передаваемого в LLM. Он решает проблему переполнения контекстного окна не путем простого обрезания, а через интеллектуальное разбиение и управляемое обогащение контекста. И, наконец, он обеспечивает надежность и непрерывность за счет явного управления состоянием и использования прозрачной архитектуры на основе рабочих процессов.
